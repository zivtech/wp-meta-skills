# Plan 011 Implementation Review

## Scope and architecture

The reviewed range makes repair certification select an explicit executor/profile adapter before generation. Plugin runtime retains the isolated standard profile. Block runtime requires a fixture-owned block name, frontend selector, and expected text, then carries that immutable assertion through the staged artifact request into a distinct `block-runtime` profile. Blueprint runtime is rejected before generation because this repository still has no generic Blueprint runtime oracle. Static execution remains available for the three named executors; runtime is never silently downgraded to static.

The block adapter builds and validates the exact emitted artifact, binds its execution proof to a disposable synthesized plugin wrapper, and runs registration, editor insertion/save, frontend selector/text, Plugin Check, WPCS, topology, identity, timing, cleanup, and manifest gates inside the inherited isolated Docker boundary. The gateway starts with the reviewed profile and fixed disposable post ID. Its only authenticated write is WordPress core's observed `POST` plus `X-HTTP-Method-Override: PUT` request to the exact plain-permalink REST route for that post with the exact `content`, `id`, and `status` body keys. Literal or broader writes, another post ID, pretty-permalink routes, and the write route in non-block profiles remain denied.

The generated-runtime CI job now provisions the exact committed block-build and acquisition-proxy images before disk admission and proves removal of its package, proxy, Compose, network, and image resources afterward. Failed sandbox commands retain only bounded, sanitized diagnostics. Generated npm/PHPUnit commands receive the private sandbox home explicitly. Plugin Check keeps WordPress sealed: its temporary content directory is created and authoritatively quarantined/validated/removed outside the WP-CLI child, while the plugin directory stays read-only.

Browser readiness is capability-based rather than delay-based. The policy waits for actionable `core/block-editor` and `core/editor` stores, verifies them again before use, and keeps post identity bound independently through the exact edit URL, gateway route/body, and frontend proof. Pinned asset size ceilings admit the exact WordPress 7.0.1 editor payload. Runtime diagnostics expose bounded gate status and failed checks, with complete Authorization, Proxy-Authorization, Cookie, and Set-Cookie values redacted in colon and equals serializations before final-tail truncation.

## Review history and dispositions

The initial drift review rejected the planned host-editor continuation because Plans 008-010 divert staged artifacts into an isolated topology that is destroyed before control returns. The plan was amended to keep the block assertion and browser proof inside that same topology. Cold review then found that the gateway admitted arbitrary numeric post IDs and could not distinguish the block profile. The implementation bound one deterministic `wp_insert_post()` import ID through the immutable request, gateway command, browser URL, REST route, and body.

Hosted proof deliberately remained red while each real incompatibility was observed. Successive exact-SHA runs exposed job-local missing build images, discarded sandbox diagnostics, a missing sandbox `HOME`, Plugin Check's write to sealed `wp-content`, cleanup ordering/stat-cache races, absent editor stores, an over-specific store-derived post-ID readiness check, truncated pinned editor assets, WordPress core's method-override transport, the plain-permalink REST route, and finally a non-WPCS-clean synthesized wrapper. Each failure received a focused regression and causal fix; no timeout, writable `wp-content`, host-browser fallback, static substitution, or broader gateway route was used to manufacture a green result.

Final review found two additional evidence defects before commit. The real WPCS regression lacked a subprocess timeout, and diagnostic traversal inspected more checks than it could emit; both are now bounded. Security review then reproduced Bearer/Basic and secondary-cookie disclosure through the new full-profile diagnostic, including `key=value` serialization and credentials longer than the final tail. Header-specific redaction now runs before slicing, and direct plus actual-sink canaries cover Authorization, Proxy-Authorization, Cookie, and Set-Cookie forms. General, security, and performance critics accepted the exact reviewed tip and its hosted evidence with no unresolved Critical, Major, or Minor findings.

The final integration scope audit also found a process failure: `block_runtime_wrapper.py`, `wp_runtime_evidence.py`, `test_runtime_image_provision.py`, `test_wp_env_network_guard.py`, and `test_wp_security_gate.py` had landed before they were added to Plan 011's declared scope. They were causally required by the hosted WPCS/workflow failures and the credential-diagnostic review, but they were not preapproved. Before this checkpoint, the ignored plan was amended with those exact narrow purposes, and a cold review confirmed that all 39 changed paths now match all 39 declared paths with no extra or missing file. The cold review accepted the corrected scope and negative space; this packet records the miss rather than treating the amendment as prior approval.
