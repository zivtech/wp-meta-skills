# WordPress Skill Improvement Research Notes - 2026-06-20

## Bottom Line

The latest WordPress evals do not support a frontier-model per-task review-quality claim. Pairwise judging, the fast answer-key diagnostic, and the adversarial answer-key diagnostic all converge on the same uncomfortable result: `zivtech_prototype` is roughly a strong few-shot prompt for Sonnet-level review tasks, not measurably better.

The current baseline default should not be Sonnet. As of the 2026-06-20 plugin-executor smoke, the repo has an isolated ChatGPT-level baseline lane through local Codex CLI: `baseline_provider: codex`, `baseline_model_policy: newest-chatgpt-level-at-run-time`, currently resolved as `gpt-5.5`. The invoke-based WordPress smoke suites now declare that policy explicitly for condition names beginning `baseline-*`. The first clean zero-shot run (`wordpress-plugin-executor-chatgpt-baseline-isolated-smoke-20260620c`) passed packet, materialization, current static artifact including `php_wpcs_shape_heuristics`, and `php-lint` gates on the simple fixture. That means simple executor-contract fixtures are also likely saturated against the newest ChatGPT-level baseline; useful skill claims need harder runtime, WPCS, variance, or modern-surface gates.

That does not make the WordPress skill suite useless. It means the useful target is narrower:

- exact WordPress API remediation and verification contracts;
- consistency and output-contract conformance across runs and authors;
- cheaper-model lift, now directionally unsupported by the first Haiku adversarial answer-key run;
- performance against a newest ChatGPT-level baseline, which is now the default comparator for invoke-based WordPress smoke baseline runs;
- executor tasks with objective oracles, such as build, WPCS, PHPUnit, wp-env, Plugin Check, block validation, and browser/editor smoke tests. The first deterministic evaluator-feedback repair loop converted a real Abilities artifact WPCS-shape failure into a ChatGPT-level repaired packet that passed static certification, WPCS, Plugin Check, plugin activation, and Abilities execution;
- modern WordPress agent surfaces, especially Abilities/MCP work, where workflow scaffolding matters more than generic review recall.

## Research Synthesis

- Anthropic's Agent Skills docs frame skills as reusable domain workflows, context, best practices, and optional scripts/templates loaded progressively. That supports compact procedural contracts over long domain lectures.
- Anthropic's agent guidance says evaluator-optimizer loops fit only when criteria are clear and iterative refinement measurably improves output. The WordPress review evals show review recall/specificity are already saturated, so a loop around those axes is unlikely to help.
- Anthropic's tool-writing guidance treats agent-facing tools as contracts between deterministic and nondeterministic systems and recommends real-world evaluations, meaningful returned context, token efficiency, and iteration against measured failures.
- OpenAI's agent guide recommends maximizing a single well-instructed agent before adding multi-agent complexity; use multi-agent splits when instructions or tool choices become overloaded. For this repo, the next split should be task-surface specific, not another generic critic layer.
- OpenAI's current eval guidance emphasizes task-specific evals, automated scoring where possible, logging real failures, and avoiding vibe-based evaluation. That maps directly to the WordPress pivot from judge-only review scoring to saved packets, materializers, artifact gates, runtime smoke, and repair prompts.
- OpenAI's skills/Codex guidance treats skills as small packages of operational knowledge with progressive disclosure: metadata first, then `SKILL.md`, then optional scripts/references/assets. That argues against stuffing more generic WordPress prose into every prompt and for compact protocols backed by reusable harness scripts.
- OpenAI's iterative Codex repair-loop cookbook makes validation failures into structured feedback for the next pass. The `repair-prompt.md` certifier artifact now follows that pattern and was verified on the Abilities/WPCS failure.
- OpenAI's agent workflow guidance puts traces/evals before optimization loops and uses `gpt-5.5` as the default agent/analysis model in its agent-improvement loop example. That supports using the newest ChatGPT-level baseline as the default comparator while keeping historical Sonnet results explicitly historical.
- OpenAI Agents SDK guardrails expose the useful pattern for this repo: tripwires should halt execution when validation fails. The WordPress certifier now halts on deterministic gates and emits a repair prompt instead of letting flawed packets become "passed by prose."
- DSPy/GEPA-style prompt optimization is relevant only if the metric reflects the desired improvement. The current objective answer-key metric found a concrete metric gap: exact API naming, not detection recall.
- DSPy Assertions are relevant at the mechanism level: assertion-driven backtracking and counterexample bootstrapping fit the WordPress artifact gates better than another hand-written review rubric.
- Reflexion, Self-Refine, and Voyager all point toward feedback from environment, execution errors, and reusable skill libraries. The WordPress suite should prioritize executable feedback loops: WPCS/PHPUnit/Playwright/wp-env/Plugin Check/Abilities discovery, rather than judge-only prose loops.
- SWE-agent's lesson is that the agent-computer interface matters: better file, test, and command interfaces can improve coding-agent outcomes. The WordPress skill edge should live in materializable packets and testable WordPress interfaces, not only in domain reminders.
- Current WordPress direction matters. `block.json` remains the canonical block registration surface; WordPress 6.7 added `wp_register_block_metadata_collection()` for multi-block plugin performance; the Interactivity API is a standard frontend interactivity path for blocks from WordPress 6.5 onward; WordPress 6.9 introduced the Abilities API; 2026 developer updates describe client-side Abilities/AI Client work aimed at browser agents and WebMCP-style integrations. A 2026-06-20 source refresh confirmed the specific surfaces the skills should name: `wp_register_ability()` on `wp_abilities_api_init`, `input_schema`, `output_schema`, `execute_callback`, `permission_callback`, `wordpress/mcp-adapter`, `mcp_adapter_init`, `mcp-adapter-discover-abilities`, `mcp-adapter-execute-ability`, `wp_ai_client_prompt()`, `wp_connectors_init`, `@wordpress/abilities`, and `@wordpress/core-abilities`.

## Design Implications

The research consensus that matters for this repo is not "add a longer prompt." ReAct-style agents improve when reasoning is interleaved with environment actions; Self-Refine and Reflexion improve outputs by feeding back concrete task results; Voyager's skill-library result points toward executable reusable behaviors plus environment feedback; Anthropic's Skills guidance explicitly says to start with representative failures and package deterministic code when reliability matters; OpenAI's agent-eval guidance separates traces for debugging from repeatable datasets/eval runs once "good" is known. The WordPress amendment should therefore prioritize runnable packet/materialization/runtime oracles and repair prompts over another prose-only critic protocol.

1. Keep the V1 frontier-review benchmark blocked. More runs at the same target mostly resample a saturated measurement.
2. Treat the new Exact API and Verification Contract as the earned V1 amendment. It directly addresses the only spread the answer-key diagnostic found. Validate it with `scripts/validate-wordpress-exact-api-contract.py`, which checks prompt contract coverage and blocks generic `expected_wordpress_apis` labels in the candidate rubrics.
3. Run model-dependence only as a new question: whether the skill lifts cheaper/weaker models toward the frontier baseline. The first Haiku adversarial answer-key run (`answerkey-haiku-adversarial-fast-20260620`) did not show broad lift, so do not back-project a cheap-model value claim without stronger new evidence.
4. Make newest ChatGPT-level baselines the prospective default for new WordPress comparisons. `evals/harness/invoke.py` now routes only condition names beginning `baseline-*` with `baseline_provider: codex` through isolated local Codex CLI, temp cwd, `--ignore-user-config`, `--ignore-rules`, and metadata sidecars, and WordPress planner/critic/executor smoke configs declare that policy. `run_wordpress_candidate_pilot.py`, `run_pairwise_pilot.py`, and `answer_key_score.py --generate` now route `baseline-*` generation through isolated local Codex (`gpt-5.5` by default) while keeping skill/upstream lanes on explicit isolated Claude prompt surfaces. Historical Claude/Sonnet result files remain historical evidence and must not be rewritten. A fresh candidate-comparison run is still required before claiming new ChatGPT-baseline results; the old absolute pilot remains historical because its artifacts were generated before this migration and its scoring is still single local Claude judge/internal-only.
5. Reframe future evaluations toward contract adherence, variance reduction, and oracle-backed code generation. The general saved-output gate is now `evals/harness/validate_wordpress_skill_output.py`, and the executor gate has three deterministic layers: saved-packet validation via `evals/harness/validate_wordpress_executor_packet.py`, packet-to-file materialization via `evals/harness/materialize_wordpress_executor_packet.py`, then generated-artifact validation via `evals/harness/validate_wordpress_artifact.py`. `evals/harness/certify_wordpress_executor_artifact.py` composes those layers into one inner-loop command and now emits a fence-safe `repair-prompt.md` on failed or blocked certifications, turning deterministic gate failures into model-readable revision prompts. A narrow local `php-lint` plus `wp-env` smoke passed on 2026-06-20, including `--wp-env-root` support for nested artifacts; `evals/harness/run_wordpress_runtime_smoke.py` now makes that lane repeatable for disposable fixtures, generated plugin artifacts, and generated block artifacts. The same harness can provision Composer-installed WPCS plus Plugin Check inside `wp-env`; the first generated modern Abilities artifact passed Plugin Check and Abilities execution but failed WPCS. The static artifact gate now catches its missing `@package` and short-array failure earlier through `php_wpcs_shape_heuristics`, and the first ChatGPT-level repair-loop run (`wordpress-plugin-executor-abilities-ai-repair-loop-20260620c` plus `generated-abilities-ai-repair-loop-full-profile-20260620c`) passed packet, materialization, static certification, `php-lint`, WPCS/PHPCS, Plugin Check, plugin activation, WordPress `7.0` `wp-env`, and `wp_get_ability()` execution. The runtime harness now also has a layered block path: `wp-env-block-runtime-smoke-20260620` passed plugin activation, block metadata, `wp-env`, and `WP_Block_Type_Registry` registration for `acme/runtime-card`; `wp-env-block-editor-smoke-20260620` then opened the post editor with Playwright and proved the same block is available in the editor-side block registry without page or console errors; `wp-env-block-editor-insert-render-smoke-20260620` inserted the block, published a post, opened the frontend permalink, and proved the server-rendered block text appears. The generated block packet `evals/suites/wordpress-block-executor/examples/smoke-wordpress-v1.materializable-packet.md` then passed saved-packet certification, materialized into block-only files, and passed `generated-block-full-profile-20260620` through a harness-synthesized wrapper plugin with disposable `npm install`, `npm run build`, WPCS/PHPCS, Plugin Check, block registration, editor insertion/save/publish, and frontend render. The generated plugin PHPUnit packet `evals/suites/wordpress-plugin-executor/examples/phpunit-wordpress-v1.materializable-packet.md` passed saved-packet certification, materialized into plugin files, and passed `generated-plugin-phpunit-full-profile-20260620` with plugin activation, artifact-local Composer-installed PHPUnit (`3 tests, 3 assertions`), WPCS/PHPCS, Plugin Check, and WordPress `7.0` `wp-env`. On 2026-06-21, the generated Interactivity block packet `evals/suites/wordpress-block-executor/examples/interactivity-wordpress-v1.materializable-packet.md` passed saved-packet certification, materialized into block-only files, and passed `generated-block-interactivity-full-profile-20260621` with metadata-aware `wp-scripts --experimental-modules` build output, built `viewScriptModule`, WPCS/PHPCS, Plugin Check, editor insertion/save/publish, frontend render, and a Playwright click/state assertion changing `context.count` from `0` to `1`. The generated deprecation block packet `evals/suites/wordpress-block-executor/examples/deprecation-wordpress-v1.materializable-packet.md` then passed saved-packet certification, materialized into block-only files, and passed `generated-block-deprecation-full-profile-20260621` with block build, WPCS/PHPCS, Plugin Check, editor loading of a legacy serialized fixture, exact migrated `content` attribute verification, current block-tree serialization, save, and frontend text proof. The generated MCP Adapter packet `evals/suites/wordpress-plugin-executor/examples/mcp-adapter-wordpress-v1.materializable-packet.md` then passed saved-packet certification, materialized into plugin files, and passed `generated-mcp-adapter-full-profile-20260621` with MCP Adapter installation, default server listing, STDIO `tools/list`, public ability discovery through `mcp-adapter-discover-abilities`, public ability execution through `mcp-adapter-execute-ability`, WPCS/PHPCS, and Plugin Check. The generated AI Client provider packet `evals/suites/wordpress-plugin-executor/examples/ai-client-provider-wordpress-v1.materializable-packet.md` then passed saved-packet certification, materialized into plugin files, and passed `generated-ai-client-provider-full-profile-20260621` with deterministic no-auth provider registration/configuration, connector registration, `wp_ai_client_prompt()`, model preference selection, generated output, WPCS/PHPCS, and Plugin Check. WPCS, Plugin Check, PHPUnit, block build, block registration, editor registry availability, fixture-level insertion/rendering, generated block-artifact insertion/rendering, generated block Interactivity behavior, generated block deprecation migration behavior, generated MCP Adapter exposure/execution, and deterministic AI Client provider-call behavior are now real feedback loops rather than aspirational verification lines; credentialed external-provider behavior remains explicit negative space.
6. Add current WordPress AI/agent surfaces to future fixtures and deterministic gates: Abilities API registration, permission callbacks/input-output schemas, MCP adapter discovery/execution, and AI Client/Connectors boundaries. These are better fit for a meta-skill than another generic plugin-security review. The Exact API contract and artifact oracle now have a static gate for these surfaces; generated Abilities execution is proven via `wp_get_ability()` in WordPress `7.0`, generated MCP Adapter exposure/execution is proven through `wp mcp-adapter serve`, and deterministic no-auth AI Client provider behavior is proven through `wp_ai_client_prompt()` plus connector/provider evidence. Credentialed external-provider behavior remains open.
