02 · the mechanism

The repair loop

Generate a materializable packet, certify it against real WordPress gates, feed the exact failures back to the model, and re-certify — until it is gate-clean or a repair budget is reached.

step 01
Generate
Model writes a materializable plugin/block/theme packet from the spec.
→
step 02
Certify
Run real WordPress gates:
  • packet contract
  • WPCS / PHPCS
  • Plugin Check
  • wp-env activation
  • Abilities / MCP / AI runtime
→
step 03 · exit
Green
WPCS-clean, Plugin Check-clean, activates in WordPress 7.0. Shippable.
↶ On failure — repair: feed the exact gate failures (errors and warnings) back to the model and regenerate. Loop to step 01.

// the generator is swappable; the gates are fixed. deterministic tooling, not an LLM judge.

04 · evidence

Cross-tier results

Early, internal, n=1 fixture (a WordPress 7.0 Abilities-API plugin). "Iterations to green" = repair passes needed to reach the full runtime gate. A single observation per tier — directional, not a benchmark.

qwen2.5-coder:32b local · q8 no — reliably reaches the runtime gate every repair iteration; the loop now absorbs transient gen timeouts (retry + last-good preservation), exposing the real ceiling — converges to a fixed-point packet that still fails real WPCS + Plugin Check

// num_ctx fixed truncation; retry + last-good preservation now absorb transient gen timeouts (latest run: 3 timeouts, 0 progress lost, 6/6 iterations certified). With the infra noise gone, the local 32B reliably reaches the runtime gate but plateaus there on real WPCS/Plugin-Check findings — a clean capability read at last.

The lever is feedback quality, not model size.

Gemini flash stalled at the repair budget until the harness was fixed to feed it the specific WPCS warnings — not just errors. With honest feedback, the cheap model converged. The same lever moved the local coder further: once the gate told it how to satisfy each verification term (not just which was missing), the local 32B cleared the entire static contract and reached real WordPress runtime checks — WPCS and Plugin Check. Then making the loop resilient — retry timed-out generations, never discard the last-good packet — stripped out the infrastructure noise: across six iterations it now reaches the runtime gate every time, but converges to a fixed-point packet it cannot push past. The real ceiling is fixing genuine findings (1 WPCS error, 6 warnings, and a Plugin Check readme issue), not infra.

05 · honest stance

What it does — and where it stops

What we claim

  • The loop gets a cheap API model to gate-clean WordPress code (gpt-5.5, gemini flash).
  • Output is WPCS-clean, Plugin Check-clean, and activates in WordPress 7.0.
  • It works across model tiers via one provider flag.
  • The harness, gates, and repair loop are the durable value.

Where it stops

  • Prompt wording isn't the lever — personas match a strong baseline.
  • Local models reliably reach the runtime gate but plateau on real WPCS/Plugin-Check findings — a model-capability ceiling, not infra.
  • Evidence is n=1 — directional, not benchmark-grade.
  • Gates measure shippability, not deep behavioral correctness.
06 · the market

Built for developers without frontier tooling

WordPress's Open Source nature means that we cannot assume that users will have access to, or opt to adopt, more expensive frontier models.

The point of this toolchain is that it doesn't require one: a cheap API model plus a deterministic gate and repair loop already ships code that passes real WordPress checks, and a local model runs the full pipeline into that same runtime tooling. The value travels down-market — which is where the market actually is.
